這不是公司的程式碼倉庫。這是一個人 + 一個長期 AI 協作者經營的業餘研究日誌。我們沒有 Jira 看板、沒有 sprint、沒有 CLA。我們有的，是對嚴謹、誠實、開源科學的共同承諾。

如果這聽起來不錯，以下是加入方式。

## 我們歡迎的貢獻

依我們渴望程度大致排序：

1. **獨立複製** 我們的實驗 — **特別是失敗的複製**。這個領域最缺的就是複製。你公開記錄的零結果，比十支喘吁吁的 YouTube 示範影片有價值。
2. **批判性審查** 我們的 SOP、誤差分析、或結論。看到缺陷請說。我們寧可你現在指出，也不要日後被期刊審稿人指出。
3. **更好的 SOP** — 更清楚的措辭、漏掉的系統誤差、相同規格但更便宜的硬體替代方案。
4. **文獻補充** — 我們應該知道但漏掉的論文，特別是俄文、日文、中文文獻。關鍵段落的翻譯是黃金。
5. **硬體設計改進** — 接線圖、BOM 替換、PCB 設計。
6. **軟體** — 分析腳本、繪圖工具、自動化黏合層。
7. **文件翻譯** 到其他語言。目前我們維護英文與繁體中文，兩者必須同步。

## 如何貢獻

### 小事 — 開 Issue

發現錯字、斷掉的連結、漏的引用、不清楚的解釋 — 直接開 Issue。不需要 template。我們會修。

### 複製實驗與資料 — 開 Discussion

若你複製了我們的實驗，無論結果為何（正面、負面、「我沒辦法讓它跑」）— 在 GitHub Discussion 「Replications」分類下發。內含：

- 你跑的是哪個實驗 / SOP 版本
- 你的硬體（若有 BOM 替換請寫）
- 你的環境（地點、溫度、濕度、任何相關因素）
- 原始資料（連到你的 repo 或直接附上）
- 你的詮釋

無論結果為何，我們會嚴肅看待每一份複製報告。

### 程式碼、文件、硬體 — 開 PR

較大的貢獻：

1. Fork 倉庫
2. 開一個語意化分支（`docs/improve-safety-section`、`analysis/add-allan-variance`、`hardware/cheaper-faraday-cage`）
3. 遵守紀律規則，特別是：
   - 每個聲稱要有引用
   - 誤差優先（沒誤差棒的數字不算數據）
   - 不用行銷語言
4. 開 PR，我們會審

我們不保證快速回應。對所有人來說這都是 side project。但我們保證誠實審查。

### 批判性審查 — 大聲說

若你認為我們在某件事上錯了，請說。開一個標題為「批判：[事項]」的 Issue 並說明你的論證。我們不會對此感到被冒犯。多數科學進展來自人們及早指出彼此的錯誤。

## 風格

- **程式碼**：Python 3.11+，用 `ruff` 格式化。建議加 type hints。
- **Markdown**：GitHub-flavored。結構化比較用表格。方程式用 LaTeX（`$inline$` 或 `$$display$$`）。
- **Commit 訊息**：祈使語氣、現在式、小寫開頭。例如 `add Allan variance to noise-floor analysis`，不是 `加了!!!`。
- **散文語氣**：嚴謹但溫暖。允許口語、偶爾自嘲。**規則**與**安全**章節保持簡潔不妥協。

## 我們不接受的

- 用行銷語言包裝的貢獻（「突破」「革命性」）
- 任何形式的專利主張。我們不申請專利。永遠不。
- 阻礙複製的閉源依賴
- 任何繞過 SAFETY.md 安全規則的東西
- 敵意或鄙視性批評。尖銳批評歡迎；惡意不歡迎。
